# IMPLEMENTATION PROMPT

# BLOG ENGAGEMENT API & EXTERNAL WEBSITE INTEGRATION

## Objective

Extend the existing Lunar CMS REST API to support **Blog Engagement Features** that can be consumed by any external website connected to Lunar CMS.

This implementation should make Lunar CMS not only a Headless CMS, but also the central source of truth for blog engagement and interaction.

The engagement features must be fully API-driven. External websites will render the UI (buttons, forms, counters, etc.), while Lunar CMS will securely store, process, and expose all engagement data.

---

# IMPORTANT

The following systems already exist.

* Authentication
* Workspaces
* API Keys
* REST API
* API Gateway
* Content API
* Blog Module
* Permissions
* API Documentation
* API Explorer
* Developer Portal

Do **NOT** rebuild them.

Inspect the existing architecture first and extend it.

Do not duplicate functionality.

---

# CORE PRINCIPLE

Lunar CMS owns the **data**, not the UI.

The external website owns the **presentation**.

Example

```text
Visitor

↓

External Website

↓

REST API

↓

Lunar CMS

↓

JSON Response

↓

External Website renders UI
```

No HTML should be returned by the API.

No frontend components should be returned by the API.

Only structured JSON.

---

Implement the following engagement modules.

1. Likes
2. Comments

4. Blog View Tracking
5. Related Posts
6. Reading Statistics
7. Optional "Powered by Lunar CMS"

Each feature must be fully database-backed and accessible through the REST API.

# MODULE 1 — LIKES

Implement a complete Like system.

Requirements

* Like a blog
* Remove like
* Retrieve like count
* Determine whether the current visitor has already liked the article
* Prevent duplicate likes from the same visitor

Support anonymous visitors and authenticated users.

Implement a visitor identifier strategy (cookie, anonymous token, or another secure approach) to reduce duplicate likes without requiring user accounts.

### API Endpoints

```http
POST   /v1/blogs/{slug}/likes
DELETE /v1/blogs/{slug}/likes
GET    /v1/blogs/{slug}/likes
```

Example response

```json
{
  "likes": 154,
  "liked": true
}
```

---

# MODULE 2 — COMMENTS

Implement a complete commenting system.

Features

* Create comment
* Reply to comment
* Nested comments (configurable depth)
* Edit (optional)
* Delete (soft delete)
* Moderation
* Approval workflow
* Spam detection hooks
* Comment count

Support

* Guest comments
* Optional authenticated comments
* Future CAPTCHA integration

### API Endpoints

```http
GET    /v1/blogs/{slug}/comments

POST   /v1/blogs/{slug}/comments

PUT    /v1/comments/{id}

DELETE /v1/comments/{id}
```

Each comment should contain

* Name
* Email (not publicly exposed)
* Website (optional)
* Avatar (optional)
* Content
* Parent Comment
* Status
* Created Date

# MODULE 4 — VIEW TRACKING

Implement article view tracking.

### Endpoint

```http
POST /v1/blogs/{slug}/view
```

Responsibilities

* Increment views
* Prevent rapid duplicate counts
* Ignore obvious bot traffic where possible
* Store timestamps
* Store referrer when available
* Store device type if available

Expose

```http
GET /v1/blogs/{slug}/stats
```

Response

```json
{
    "views":5230
}
```

# MODULE 8 — FEATURE FLAGS

The API should expose available features.

Example

```json
{
  "features": {
    "likes": true,
    "comments": true,
    "socialShare": true,
    "relatedPosts": true,
    "poweredBy": true,
```

This allows external websites to dynamically render only the enabled functionality.

Feature flags should be configurable per workspace.
