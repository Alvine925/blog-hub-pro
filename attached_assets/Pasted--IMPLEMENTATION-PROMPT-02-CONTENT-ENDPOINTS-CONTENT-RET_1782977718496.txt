Reuse existing repositories, services, types and database queries whenever possible.

---

# Objective

Implement the content retrieval layer of the REST API.

The API should allow external applications to securely retrieve content stored inside the existing CMS.

Every request must:

* Authenticate
* Resolve Workspace
* Validate permissions
* Return only published content (unless Secret Key permissions allow otherwise)
* Return standardized JSON

---

# Request Lifecycle

Every request follows this flow:

```text
External Website
        │
GET /v1/blogs
        │
Bearer API Key
        │
api-gateway
        │
validate-api-key
        │
resolve-workspace
        │
validate-permissions
        │
content-router
        │
query existing CMS tables
        │
transform response
        │
log request
        │
return JSON
```

No endpoint should query Supabase directly from the client.

---

# Edge Function

Create (if missing)


# Blogs

## Get All Blogs

```
GET /v1/blogs
```

Returns only published blogs.

Supports

page

limit

sort

order

search

category

tag

author

featured

from

to

Response

```json
{
  "success": true,
  "data": [],
  "meta": {
      "page":1,
      "limit":20,
      "total":125,
      "totalPages":7
  }
}
```

---

## Get Blog By Slug

```
GET /v1/blogs/:slug
```

Never expose internal IDs.

Lookup should always use slug.

If not found

# External Website Example

Example

```typescript
const response = await fetch(
"https://api.yourdomain.com/v1/blogs?page=1&limit=10",
{
headers:{
Authorization:`Bearer ${API_KEY}`
}
})

const data=await response.json()
```

No Workspace ID required.

No Project ID required.

No Collection ID required.

The API Key determines everything.

# Acceptance Criteria

The implementation is complete when:

✓ All content endpoints authenticate through the API Gateway.

✓ Every endpoint resolves the correct workspace automatically.

✓ Published content is returned correctly.

✓ Slug-based retrieval works for all supported content types.

✓ Pagination, filtering, sorting and search function correctly.

✓ Internal database fields are never exposed.

✓ All responses follow the standard response contract.

✓ Existing CMS functionality remains unchanged.

✓ Content retrieval is optimized for performance and ready for external websites, mobile apps and future SDKs.
