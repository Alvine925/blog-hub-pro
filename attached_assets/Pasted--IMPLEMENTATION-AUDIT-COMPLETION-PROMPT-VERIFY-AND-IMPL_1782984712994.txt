# PHASE 2 — VERIFY THE COMPLETE REQUEST FLOW

Verify that every incoming REST API request follows this exact lifecycle.

```text
External Website

↓

REST Endpoint

↓

API Gateway

↓

Authentication

↓

API Key Validation

↓

Workspace Resolution

↓

Permission Validation

↓

Rate Limiting

↓

Request Validation

↓

Business Logic

↓

Database Query

↓

Response Filtering

↓

JSON Response Builder

↓

Return Response
```

If any step is missing, implement it.

---

# STEP 1 — API Gateway

Verify that every request first enters a centralized API layer.

There should not be duplicated authentication logic across endpoints.

If a centralized gateway or middleware does not exist, create one.

Responsibilities include:

* Receive incoming requests
* Parse headers
* Authenticate
* Validate API Key
* Resolve workspace
* Apply rate limiting
* Validate permissions
* Pass request to endpoint

All REST endpoints must use this shared flow.

---

# STEP 2 — API KEY VALIDATION

Verify that every request validates the supplied API Key.

Expected request format

```http
Authorization:
Bearer lunar_pk_xxxxxxxxx
```

Validation must include

* Key exists
* Key is active
* Key has not expired
* Key belongs to an existing workspace
* Key has required scopes
* Key is not revoked
* Key is not suspended
* Workspace subscription permits API access

If any validation is missing, implement it.

No endpoint should bypass API Key validation.

---

# STEP 3 — WORKSPACE RESOLUTION

Verify that the workspace is resolved automatically from the API Key.

The client must never send

* workspace_id
* tenant_id
* collection_id
* organization_id

The API Key alone must determine which workspace owns the request.

If endpoints currently require workspace identifiers from the client, refactor them to use automatic workspace resolution.

---

# STEP 4 — PERMISSION CHECKS

Verify that permissions are enforced before business logic executes.

Examples

A key with

* blogs.read

must not access

* events.write

Permissions should support fine-grained scopes such as

* blogs.read
* blogs.write
* pages.read
* pages.write
* events.read
* events.write
* media.read
* media.write
* search.read

Requests without the required scope must return

```http
403 Forbidden
```

---

# STEP 5 — RATE LIMITING

Verify that rate limiting is implemented before database access.

Rate limiting should support

* Requests per minute
* Requests per hour
* Requests per day

The limits should be configurable.

When exceeded

Return

```http
429 Too Many Requests
```

Include standard rate-limit headers where appropriate.

---

# STEP 6 — REQUEST VALIDATION

Verify that

* Query parameters
* Route parameters
* Request body
* Pagination values
* Search parameters
* Filters

are validated before execution.

Reject invalid requests before querying the database.

---

# STEP 7 — DATABASE ACCESS

Verify that every database query is automatically filtered by the resolved workspace.

Example

Incorrect

```sql
SELECT *
FROM blogs;
```

Correct

```sql
SELECT *
FROM blogs
WHERE workspace_id = resolved_workspace
AND status = 'published';
```

No query should ever expose another workspace's data.

Audit every endpoint for tenant isolation.

---

# STEP 8 — RESPONSE FILTERING

Before returning data

Verify that internal fields are removed.

Never expose

* workspace_id
* owner_id
* internal_notes
* draft_content
* deleted_at
* internal_status
* audit_fields
* private_metadata
* database IDs that should remain internal

Only expose public API fields.

If response filtering does not exist, implement a centralized response transformer.

---

# STEP 9 — RESPONSE BUILDER

Verify that every endpoint returns a consistent response structure.

Successful responses should follow one standard.

Example

```json
{
  "success": true,
  "data": {},
  "meta": {},
  "links": {}
}
```

Errors should also use one standard structure.

Example

```json
{
  "success": false,
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid API Key"
  }
}
```

If different endpoints return different formats, standardize them.

---

# STEP 10 — ERROR HANDLING

Verify centralized handling for

* 400
* 401
* 403
* 404
* 409
* 422
* 429
* 500

Error responses should be predictable and documented.

Never expose stack traces or internal database information.

---

# STEP 11 — LOGGING

Verify that API requests are logged.

Each request should record

* Timestamp
* Endpoint
* Method
* Workspace
* API Key ID (not the raw key)
* Status Code
* Response Time
* IP Address (where available)
* User Agent

Sensitive information must never be logged.

---

# STEP 12 — CACHING

Verify caching for read-only endpoints.

Suitable examples include

* Homepage
* Blogs
* Pages
* Events
* Categories

Cache should be invalidated automatically whenever related content is published, updated or deleted.

---

* Mass Assignment

* Cross-workspace data leakage
* Invalid pagination values
* Oversized payloads
* Excessive request rates
* Invalid HTTP methods

Implement missing protections.
