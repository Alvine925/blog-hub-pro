# Objective

Transform the existing CMS into a Headless CMS by implementing a secure REST API layer.

External applications should be able to retrieve content using an API Key without ever connecting directly to Supabase.

Every request must pass through Edge Functions where authentication, authorization, logging, analytics and validation occur.

The frontend should never expose the Supabase Service Role Key.

---

# How External Websites Will Work

Assume a customer owns:

https://myagency.com

Inside Lunar CMS they create content.

The Workspace already exists.

The user generates an API Key.

Example:

```text
pk_live_R8Y2KJ29M4...
```

Inside their Next.js website they install the SDK (to be implemented later) or call the REST API directly.

Example request:

```http
GET https://api.yourdomain.com/v1/blogs

Authorization:
Bearer pk_live_xxxxxxxxxxxxx
```

The external website DOES NOT send:

* workspace_id
* project_id
* collection_id

Only the API Key.

The API Key should be enough to identify the workspace.

# API Authentication Flow

When a request arrives:

Step 1

Read

Authorization Header

Reject immediately if missing.

Expected format:

Bearer pk_live_xxxxxxxxx

or

Bearer sk_live_xxxxxxxxx

---

Step 2

Extract the token.

Never trust the client.

Validate format.

Reject malformed tokens.

---

Step 3

Hash the incoming token using the same hashing algorithm used during storage.

Never compare plaintext.

---

Step 4

Lookup the hashed key in the database.

If not found

Return

401 Unauthorized

---

Step 5

Check

Active

Not Expired

Not Revoked

Workspace Exists

Workspace Active

---

Step 6

Resolve Workspace

Example

```text
API Key

↓

Workspace ID

↓

Workspace

↓

Existing Collections

↓

Existing Content
```

The client should never know the Workspace ID.

Everything is automatic.

---

Step 7

Check Permissions

Example

Publishable Key

Can:

Read Blogs

Read Pages

Read Media

Read Events

Cannot:

Create Blogs

Delete Blogs

Manage Users

Generate API Keys

Secret Keys can perform write operations.

---

Step 8

Apply Rate Limiting

Free

100/hour

Starter

1000/hour

Pro

10000/hour

Business

Configurable

---

Step 9

Continue to Endpoint

---

Step 10

Log Request

Every request.

Even failed requests.

# API Key Generation

Create an Edge Function

generate-api-key

Responsibilities

Generate cryptographically secure random token.

Example

pk_live_xxxxxxxxxxxxxxxxx

or

sk_live_xxxxxxxxxxxxxxxxx

Never store the raw key.

Store only

SHA-256 hash

Return plaintext only once.

If the user loses it they must generate another.

Display warning.

"This key cannot be viewed again."

Exactly like GitHub.

# API Response Standard

Every response must follow one format.

Success

```json
{
    "success": true,
    "data": {},
    "meta": {}
}
```

Errors

```json
{
    "success": false,
    "error": {
        "code": "INVALID_API_KEY",
        "message": "The supplied API key is invalid."
    }
}
```

Never expose stack traces.

Never expose SQL errors.

Never expose internal IDs.

---

# Security Requirements

Never expose:

Service Role Key

OpenAI Keys

Gemini Keys

Workspace IDs

Internal Database IDs

Supabase Secrets

Never store raw API Keys.

Never trust frontend validation.

Validate every request server-side.

Apply Row Level Security to new tables.

Use constant-time comparison when checking hashes.

Reject requests with invalid Authorization headers before querying content.

# External Website Developer Experience

The goal is to make integration extremely simple.

Example:

```typescript
const response = await fetch(
"https://api.yourdomain.com/v1/blogs",
{
headers:{
Authorization:`Bearer ${process.env.NEXT_PUBLIC_LUNAR_KEY}`
}
}
)

const blogs=await response.json()
```

The developer should not need to know:

Workspace ID

Collection IDs

Database IDs

Project IDs

The API Key should resolve everything automatically.

# Acceptance Criteria

This implementation is complete only when:

✓ Users can generate secure API Keys.

✓ Keys are stored only as hashes.

✓ API Gateway authenticates every request.

✓ Workspace resolution is automatic.

✓ External applications authenticate using only Bearer tokens.

✓ Unauthorized requests return 401.

✓ Invalid keys never reveal why validation failed.

✓ Every request is logged.

✓ API usage metrics are updated automatically.

✓ Existing CMS functionality remains untouched.

✓ The REST API foundation is production-ready and can support future endpoints without architectural changes.
